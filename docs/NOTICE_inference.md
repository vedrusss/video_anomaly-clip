# NOTICE: Инференс

## Коротко

Инференс в этом проекте работает как конвейер из трёх отдельных стадий:

1. `preprocess` - чтение видео, семплирование кадров, resize/crop/normalization.
2. `encoding` - прогон выбранных кадров через CLIP visual encoder (`ViT-B/16`) и построение feature-векторов.
3. `classify` - прогон feature-векторов через `AnomalyCLIP` поверх загруженного чекпоинта.

Под `CLIP visual encoder ViT-B/16` здесь имеется в виду визуальная часть OpenAI CLIP, которая превращает каждый кадр в компактный embedding-вектор. Именно эти embeddings потом подаются в верхнюю часть AnomalyCLIP.

## Что важно помнить

- Базовая конфигурация в инференсе жёстко завязана на `num_segments = 32` и `seg_length = 16`.
- Это даёт базовую длину входа `32 * 16 = 512`.
- Без новой тренировки или хотя бы серьёзной адаптации архитектуры менять эту связку нельзя.
- В текущем коде эти значения захардкожены в `inference/common.py`, а не вынесены в CLI.

Иными словами, `512` - это не просто размер тензора, а структурная длина последовательности, на которую завязаны reshape-операции, temporal-модуль и часть логики loss.

## Структура таймингов

На вашем прогоне было видно:

- `preprocess` - около `1.27s`
- `encoding` - около `3.43s`
- `classify` - только несколько миллисекунд, порядка `5.5 ms` в ваших замерах

Это было для видео `G4GVK6ZQ-336-30005216_4N72787A6CFA7_1514327102059.mp4` и чекпоинта `checkpoints/shanghaitech/last.ckpt`.

Практический вывод:

- Сам CLIP-энкодер даёт основную часть времени на инференсе.
- Этап `classify` относительно лёгкий, потому что он работает уже с готовыми feature-векторами.
- Паддинг до 512 делает дорогим даже хвост из повторяющихся кадров.

Практический смысл этих цифр такой: основной bottleneck сейчас не в верхнем классификаторе, а в прогоне кадров через CLIP visual encoder.

## Семплирование кадров

- Сейчас raw video семплируется через `fps`, по умолчанию `fps=8`.
- Это не поиск ключевых кадров, а простое равномерное снижение частоты.
- Если видео длинное, а интересные события локальны, такой подход может быть не самым информативным.

Параметр `fps=8` означает, что из исходного потока кадров берётся примерно 8 кадров в секунду, если source video имеет более высокий FPS. Это грубое семплирование, а не семантический отбор кадров.

### Практический недостаток

Если надо всё равно довести вход до 512, то более разумно было бы:

- сначала выбирать более информативные кадры;
- потом уже добирать длину до нужной кратности.

Сейчас же логика проще:

- выбрать кадры по FPS;
- добить до `512` повторением последнего кадра.

## Паддинг

Сейчас паддинг делается повторением последнего кадра.

Это корректно для формы тензора, но не оптимально по вычислениям:

- CLIP-энкодер отдельно прогоняет и эти дубли;
- на коротких видео это может быть значительная часть работы.

То есть часть вычислений может уходить не на новую информацию, а на повтор одного и того же последнего кадра, который добавляется только ради выравнивания длины.

### Более эффективная идея

Без изменения логики модели можно было бы:

- один раз закодировать последний кадр;
- затем паддить уже feature-вектором, а не повторно гонять изображение через CLIP.

Это сохранило бы поведение модели, но уменьшило бы лишнюю работу на encoding.

Идея здесь простая: `image -> CLIP` для хвостового кадра делается один раз, а дальше повторяется уже готовый feature, а не сам кадр.

## Примечание про `ncentroid`

- Если рядом с чекпоинтом лежит `ncentroid.pt`, инференс использует его.
- Если нет, он может посчитать centroid на лету из текущего видео.
- Для новых видео это поддерживается, но сохранённый centroid обычно стабильнее.

`ncentroid` - это средний вектор нормального пространства признаков, то есть усреднённый embedding нормальных кадров или features. Он нужен для центрирования image/text features перед сравнением.

Практически это означает, что инференс не обязан падать без отдельного `ncentroid.pt`, но сохранённый centroid обычно даёт поведение ближе к train setup.
